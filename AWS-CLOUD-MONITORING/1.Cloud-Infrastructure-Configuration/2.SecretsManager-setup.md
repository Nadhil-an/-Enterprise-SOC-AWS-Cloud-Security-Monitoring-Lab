
# AWS Secrets Manager Event Generation and Monitoring

## Overview

This setup demonstrates how AWS Secrets Manager was integrated into the SOC monitoring lab to generate real-world AWS security events and validate end-to-end monitoring visibility in Splunk.

The generated Secrets Manager events are monitored using:

- AWS CloudTrail
- Amazon S3
- Amazon SQS
- Splunk Enterprise

This validates that sensitive AWS API activities are successfully captured and ingested into the SIEM pipeline.

---

# Objective

The purpose of this setup is to:

- Generate AWS management events
- Monitor secret creation activity
- Validate CloudTrail event logging
- Verify S3 log delivery
- Verify SQS notification delivery
- Confirm Splunk log ingestion
- Simulate real-world cloud monitoring use cases

---

# Architecture Flow

AWS Secrets Manager → CloudTrail → S3 Bucket → SQS Queue → Splunk Enterprise

---

# AWS Services Used

| Service | Purpose |
|---|---|
| AWS Secrets Manager | Generate secret management events |
| AWS CloudTrail | Record API activity |
| Amazon S3 | Store CloudTrail logs |
| Amazon SQS | Notify Splunk about new logs |
| Splunk Enterprise | Analyze and monitor events |

# Step 5 — Generate Secret Access Events

After creating the secret:

- Open the created secret
- View stored values
- Access the secret
- Retrieve secret information

These activities generate CloudTrail events such as:

| Event Name | Purpose |
|---|---|
| CreateSecret | Secret creation |
| GetSecretValue | Secret access |
| DescribeSecret | Secret metadata access |
| ListSecrets | Secret enumeration |

# Step 6 — Verify CloudTrail Event Logging

Navigate to:

AWS Console → CloudTrail → Event History

Search for:

- CreateSecret
- GetSecretValue
- DescribeSecret

Verify:

- Event source
- Username
- Event time
- AWS region
- Resource accessed

This confirms CloudTrail successfully captured the activity.

---

# Step 7 — Verify Logs in S3 Bucket

Navigate to:

AWS Console → S3 → CloudTrail Log Bucket

Verify:

- AWSLogs folders created
- Region-specific log folders available
- New CloudTrail log files generated

Example regions:

- us-east-1
- us-east-2
- ap-southeast-2

This confirms CloudTrail successfully delivered logs to S3.

---

# Step 8 — Verify SQS Notification Delivery

Navigate to:

AWS Console → SQS → Queue

Verify:

- Queue receives new notifications
- Message count increases
- S3 event notifications are delivered

This confirms the S3-to-SQS pipeline is functioning correctly.

---

# Step 9 — Verify Events in Splunk

Open Splunk Search and query:

```spl
index=default sourcetype=aws:cloudtrail
```

Search for:

```spl
eventName=CreateSecret
```

or

```spl
eventName=GetSecretValue
```

Verify:

- Secret creation events visible
- User activity logged
- AWS API activity searchable
- Timestamps correctly indexed

This confirms successful end-to-end SIEM ingestion.

# Detection Use Cases

The following detections can be created from Secrets Manager logs:

| Detection | Security Purpose |
|---|---|
| Excessive secret access | Detect credential harvesting |
| Unauthorized secret retrieval | Detect suspicious activity |
| Secret enumeration | Detect reconnaissance |
| Cross-region secret access | Detect anomalous behavior |
| Root account secret access | Detect privileged misuse |
